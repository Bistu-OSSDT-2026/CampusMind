## 一、文档概述

### 1.1 文档目的

本文档为「校园学业规划助手」（产品代号 GuardGPA，对外传播名「防挂科神器」）的产品需求规格说明，用于指导产品设计、技术研发、测试验收及后续迭代。文档界定产品边界、明确核心场景、定义功能模块与交互流程，并给出优先级与分期落地计划。

### 1.2 产品定位

GuardGPA 是一款聚焦于大学生「学业规划」单一场景的智能助手。产品刻意摒弃全场景校园服务的庞杂形态，只解决两个高频痛点：

1. 课程与考试信息碎片化导致的「不知道下一步该干什么」；
2. 复习无章法导致的「考前焦虑与低效突击」。

> **一句话定位：让你不挂科、复习有章法。**

> **核心信念：** 做窄不做宽。把「防挂科」这一个场景打穿，比堆砌十个浅功能更有价值。所有功能取舍的唯一标准是：**是否直接提升学习效率、降低挂科风险**。

### 1.3 术语与缩写

| 术语 / 缩写 | 全称 / 含义 | 说明 |
|-------------|-------------|------|
| PRD | Product Requirements Document | 产品需求文档 |
| LLM | Large Language Model | 大语言模型，本产品复习计划生成核心 |
| DDL | Deadline | 截止时间，含作业截止与考试时间 |
| Tool A / B / C / D / E | 五类功能工具 | 课表 / 死线 / 计划生成 / 进度追踪 / 学情复盘 |
| D-N | 距离考试 N 天 | 倒计时标记，如 D-3 表示距考试 3 天 |
| 早报 / 晚报 | 每日定时推送 | 8:00 早报课程与任务，22:00 晚报打卡复盘 |
| 艾宾浩斯曲线 | 遗忘曲线 | 据知识点遗忘规律安排 1/2/4/7 天回顾 |

### 1.4 读者对象

- **产品经理、交互设计师**：理解产品边界与功能优先级。
- **研发工程师**：明确模块划分、接口契约与数据模型。
- **测试工程师**：依据验收标准制定用例。
- **项目负责人 / 投资人**：评估产品价值与落地节奏。

## 二、背景与问题分析

### 2.1 用户痛点

大学生在学业规划环节长期处于「被动救火」状态，具体表现为三类痛点：

- **信息碎片化**：课程表、作业截止时间、考试安排分散在教务系统、课程群、班级通知、纸质课表等多处，缺乏统一视图，漏看一条通知即可能导致作业零分。
- **时间无规划**：即便知道考试日期，多数学生仍倾向于考前 1–2 天突击，无法将复习任务合理拆解到考前多天，导致知识点覆盖不全。
- **复习无章法**：缺少结构化的复习路径与进度追踪，「打开书不知道从哪看起」「看完一章没有反馈」，复习沦为低效浏览。

### 2.2 现有方案不足

市场上校园类应用普遍走「大而全」路线，将社交、二手交易、后勤查询、资讯推送混在一起，导致产品重心分散，学业场景被边缘化。学生真正需要的是一把「锋利的刀」，而非一个塞满工具的杂物间。**专注、克制、工具属性强**，是本产品区别于竞品的核心差异点。

### 2.3 核心机会

- **大模型能力成熟**：LLM 已能基于考试大纲 / 教材目录生成结构化、可执行的复习计划，这是传统规则引擎难以实现的。
- **闭环可编排**：考试日期、课表空隙、复习计划、进度反馈四者可形成自动化闭环，提供远超「备忘录」的体验。
- **用户高频高粘性**：学业是学生的刚需，考前一个月为高频使用期，具备良好的留存与口碑传播基础。

## 三、目标用户与使用场景

### 3.1 目标用户画像

| 维度 | 用户 A：拖延型 | 用户 B：焦虑型 | 用户 C：效率型 |
|------|---------------|---------------|---------------|
| 年级 | 本科一、二年级 | 本科二、三年级 | 本科三、四年级 / 研究生 |
| 特征 | 日常不规划，考前突击 | 想复习但不知从何入手 | 有规划但缺工具辅助 |
| 痛点 | 忘记 DDL、时间不够 | 复习无结构、焦虑 | 手工排计划繁琐 |
| 核心诉求 | 强力倒计时提醒 | 生成可执行计划 | 自动避开课表排期 |
| 使用频率 | 考前 2 周集中使用 | 考前 1 月持续使用 | 全学期周期性使用 |

### 3.2 核心使用场景

**场景一：日常课表与任务查询**
用户在课前、课间快速查询「下节课是什么、在哪上」，以及「今天有哪些作业要交」。产品提供秒级响应，无需翻找课表或聊天记录。

**场景二：死线倒计时提醒**
用户登记作业或考试时间后，产品在 D-7、D-3、D-1、D-0 四个节点阶梯式推送提醒，语气逐级加重，迫使用户提前行动而非临阵磨枪。

**场景三：复习计划生成与执行**
用户表达复习需求（如「帮我复习高数」）后，产品结合考试日期倒推、课表空隙、考试大纲，生成结构化多日复习计划，并在执行阶段每日打卡、动态调整。

**场景四：考前冲刺与考后复盘**
考前一天推送「考前速记清单」；考后引导用户对本次复习效果做轻量复盘，沉淀薄弱知识点，指导下轮计划生成。

## 四、产品目标与成功指标

### 4.1 产品目标

- **短期（3 个月）**：完成核心闭环（课表 + 死线 + 计划生成 + 打卡），在 1–2 所高校完成种子用户验证。
- **中期（6 个月）**：接入艾宾浩斯复习调度，覆盖完整学期使用周期，月活跃用户留存率达 40% 以上。
- **长期（12 个月）**：建立学情沉淀，成为用户全学业的「规划中枢」，形成口碑传播。

### 4.2 成功指标

| 类别 | 指标 | 目标值 | 说明 |
|------|------|--------|------|
| 北极星 | 周复习计划完成率 | ≥ 60% | 生成的计划中按日完成的任务比例 |
| 活跃 | 考前 2 周周活跃用户 | 种子校 ≥ 1000 | 核心使用窗口的渗透 |
| 留存 | 次月留存率 | ≥ 40% | 体现持续陪伴价值 |
| 效果 | 用户主观挂科风险降低 | ≥ 70% 认同 | 问卷自评 |
| 效率 | 计划生成满意度 | ≥ 4.0 / 5.0 | 计划可用性评分 |

## 五、产品架构与核心闭环

### 5.1 核心闭环（产品的灵魂）

> **本产品最关键的判断**：课表（A）、死线（B）、计划生成（C）三者绝非孤岛，真正的价值在于**编排**。

```
考试/作业日期(B) ──(倒推)──▶ 复习计划(C) ──(避开课表A)──▶ 每日任务
      ▲                                              │
      └────────────── 进度反馈(D) + 遗忘曲线调整 ◀──────┘
```

缺少中间联动，C 生成的计划可能让用户「今晚复习 3 小时」，而用户今晚恰好满课。因此**联动不是可选项，而是核心能力**。闭环的起点是死线、终点是掌握度提升，中间四类工具协同运作。

### 5.2 功能模块总览

| 编号 | 模块 | 类型 | 优先级 | 所属阶段 |
|------|------|------|--------|----------|
| A | 课表查询与导入 | 工具 | P0 | V1.0 |
| B | 死线登记与倒计时 | 工具 | P0 | V1.0 |
| C | 复习计划生成 | LLM 核心 | P0 | V1.0 |
| D | 进度打卡与遗忘曲线 | 工具 + LLM | P1 | V1.1 |
| E | 学情复盘 | LLM 核心 | P2 | V2.0 |
| — | 定时推送（早报 / 晚报 / 阶梯提醒） | 确定性逻辑 | P0 | V1.0 |
| — | 边界识别与拒答引导 | 对话策略 | P0 | V1.0 |

### 5.3 信息架构

产品以单聊对话为主入口，辅以「今日面板」聚合视图：

- **今日面板（首页）**：今日课程 · 今日死线 · 今日复习任务 三合一
- **对话入口**：自然语言交互，自动路由到对应工具
- **课表视图**：周视图 + 单日详情
- **死线列表**：按 D-N 排序，支持手动新增 / 标记完成
- **计划中心**：当前执行中的复习计划及进度
- **复盘记录**：历次考试复盘归档

## 六、功能需求详述

### 6.1 Tool A — 课表查询与导入

**功能描述**：支持用户查询当前及未来课程信息，并提供低门槛的课表录入能力。课表数据同时作为复习计划排期的「时间约束」输入。

#### 录入方式（按优先级）

| 方式 | 门槛 | 实现复杂度 | 优先级 | 说明 |
|------|------|-----------|--------|------|
| 截图 OCR 识别 | 低 | 中 | P0 | 主推；用户上传教务系统截图自动解析 |
| 手动快速录入 | 中 | 低 | P0 | 兜底；表单逐条录入 |
| 教务系统导入 | 低 | 高 | P2 | 锦上添花；各校适配成本高，后置 |

> **落地提示**：「调用课表接口」的前提是数据已存在，而学生最痛的恰是录入。若不解决 onboarding，Tool A 在演示阶段基本无法跑通。**截图 OCR 是最低门槛的破局点**。

#### 查询能力

- 自然语言查询：「下节课是什么」「今天还有几节课」「在哪上课」
- 输出字段：课程名称、上课地点、起止时间、授课教师（可选）
- 冲突检测：录入死线或生成计划时，自动标注与课程的时间冲突

#### 接口规约

| 项 | 内容 |
|----|------|
| 输入 | 自然语言查询 / 日期范围 / 「下节课」等时间指代 |
| 输出 | 课程名称、地点、时间、教师；按时间升序排列 |
| 异常 | 课表未导入 → 引导录入；查询无结果 → 返回最近一节课程 |

---

### 6.2 Tool B — 死线登记与倒计时

**功能描述**：支持用户登记作业截止时间与考试时间，并按确定性时间表阶梯式推送倒计时提醒。

#### 登记字段

- **类型**：作业 / 考试 / 其他学业任务
- **科目**：关联具体课程（若已在课表中）
- **截止时间**：精确到日期，考试精确到日期 + 时段
- **权重（可选）**：标注重要性，影响提醒强度

#### 阶梯提醒规则（确定性逻辑）

> 推送采用**确定性时间表**，而非模糊的「到达时间点」。语气随距离临近逐级加重。

| 节点 | 触发时机 | 语气强度 | 推送内容 |
|------|---------|---------|---------|
| D-7 | 距截止 7 天 8:00 | 温和提示 | 「还有一周，建议开始规划。」+ 建议生成复习计划 |
| D-3 | 距截止 3 天 8:00 | 中度提醒 | 「剩 3 天，今日复习任务已排好。」|
| D-1 | 截止前 1 天 20:00 | 强提醒 | 「明天截止，今晚必须完成。」+ 精确剩余时间 |
| D-0 | 截止当天 7:00 / 前 2h | 最高强度 | 「今日截止！」+ 任务详情 |

---

### 6.3 Tool C — 复习计划生成（LLM 核心）

**功能描述**：当用户表达复习需求（如「帮我复习高数」「下周考线代怎么办」）时，结合考试日期、课表空隙、考试大纲或通用教材目录，由 LLM 生成结构化的多日复习计划大纲。

#### 输入要素

- 考试科目与日期（来自 Tool B 或用户当场指定）
- 可用复习时段（来自 Tool A 的课表空隙反推）
- 考试大纲 / 教材目录（用户提供截图，或调用通用版本）
- 用户自评掌握度（可选，影响重点分配）

#### 输出结构

计划以「天」为粒度，每天包含目标、知识点、预计时长、对应时段：

```
【高数冲刺 D-4 ~ D-0】
D-4（今）：极限与连续（基础） + 习题 P132
D-3（明）：导数与微分
D-2（周三）：中值定理（薄弱，2 天后回顾）
D-1（周四）：积分综合 + 错题重做
D-0（周五）：早 7:00 推送「考前 30 分钟速记」
```

#### 生成约束

- 必须避开课表中的上课时段，仅填充可用空隙
- 每天总复习时长不超过用户设定上限（默认 4 小时）
- 薄弱知识点安排更高权重与回顾节点
- 考前一天预留「速记 + 错题」，不安排新内容

---

### 6.4 Tool D — 进度打卡与遗忘曲线（V1.1）

**功能描述**：计划生成后必须有追踪闭环，否则计划形同废纸。Tool D 提供每日轻量打卡，并依据艾宾浩斯遗忘曲线对薄弱知识点安排 1/2/4/7 天回顾。

#### 打卡交互

- 每日 22:00 主动询问：「今天复习任务完成情况？」
- 用户一句话反馈：如「微积分第三章看完了，例题有点吃力」
- 系统标记完成度 + 提取薄弱点 + 调整后续计划节奏

#### 遗忘曲线调度

| 回顾次序 | 距上次学习 | 目的 |
|---------|-----------|------|
| 第 1 次 | 1 天后 | 短期巩固 |
| 第 2 次 | 2 天后 | 强化记忆 |
| 第 3 次 | 4 天后 | 对抗遗忘拐点 |
| 第 4 次 | 7 天后 | 转入长期记忆 |

---

### 6.5 Tool E — 学情复盘（V2.0）

**功能描述**：考完一次试后，引导用户做一次轻复盘：哪科计划没完成、哪些知识点反复薄弱、下次如何调整。这是把「一次性工具」升级为「长期陪伴」的关键，且不越出学业边界。

- **复盘维度**：计划完成率、薄弱知识点清单、时间投入分布
- **输出**：下次计划生成的个性化建议（如「该用户计算题耗时偏长，下次预留更多练习」）

---

### 6.6 确定性推送（早报 / 晚报）

将推送做成确定性时间表，确保用户每天有两个稳定的「产品触点」，而非随机打扰。

| 时间 | 类型 | 内容 |
|------|------|------|
| 8:00 | 早报 | 今日课程 + 今日死线 + 今日复习任务 三合一 |
| 按阶梯表 | 阶梯提醒 | D-7 / D-3 / D-1 / D-0 倒计时 |
| 22:00 | 晚报 | 今日复习打卡情况 + 明日预告 |

---

### 6.7 边界识别与拒答引导

**设计原则**：硬拒绝会打断体验（学生随口问「今天食堂啥菜」被甩脸，感受很差）。采用「识别 → 拒绝 → 引导回学业」的柔性策略。

**处理流程**：

```
识别到非学业意图
   ↓
一句话礼貌拒绝
   ↓
顺手把话题拉回当前最紧迫的死线 / 任务
```

> **示例**：用户「今天学校食堂有啥好吃的？」→ 回复「我只管帮你防挂科～对了，你高数作业明晚 23:59 截止，现在还剩 32 小时。」

## 七、对话交互设计

### 7.1 交互风格

- 极简、直接、工具感强，零寒暄
- 避免过度拟人化，不使用「亲爱的」「小主」等称呼
- 指令模糊时，只追问「哪门课 / 哪天 / 截止何时」，不发散

### 7.2 意图路由

| 用户表达 | 识别意图 | 路由工具 |
|---------|---------|---------|
| 下节课是什么 / 在哪上课 | 课表查询 | Tool A |
| 设个提醒 / X 号考高数 | 死线登记 | Tool B |
| 帮我复习高数 / 快炸了 | 计划生成 | Tool B + A + C 联动 |
| 今天看完了 / 有点吃力 | 进度反馈 | Tool D |
| 考完了 / 这次怎么样 | 复盘 | Tool E |
| 近期任务 / 这周有啥 | 聚合查询 | B + A 汇总 |
| 非学业内容 | 越界 | 边界拒答引导 |

### 7.3 多工具编排示例（展示核心闭环价值）

下例展示一次合成多工具的对话，而非逐个工具应答。这是「编排」相对「孤岛」的价值所在。

```
用户：周五考高数，还有一堆作业，快炸了
助手：已记录：高数考试 周五(D-4)。
```

## 八、边界与约束

### 8.1 明确不做（防止范围蔓延）

- 社交动态、好友圈、匿名墙
- 二手交易、跳蚤市场
- 后勤查询：食堂、校车、天气、宿舍
- 情感陪伴、闲聊、心理咨询
- 多人协作、班级群组功能

### 8.2 行为约束

- 绝不回答与学业无关的问题，但采用柔性引导而非硬拒绝
- 回复风格简洁直接，工具属性，避免寒暄
- 指令模糊时，仅引导用户确认具体考试科目或时间，不发散
- 所有提醒推送必须服务于「提高学习效率」目标

### 8.3 合规与隐私

- 教务系统账号密码仅用于本地导入，不存储于服务端
- 课表、死线等数据归属用户，支持一键导出与删除
- 复习计划生成调用 LLM 时，不回传可识别个人身份信息

## 十二、风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 课表录入门槛高，用户流失在 onboarding | 高 | 截图 OCR 优先；提供示例引导；3 条课表内可手动快速录入 |

| 提醒打扰导致关闭通知 | 中 | 确定性时间表（仅早晚 + 阶梯），可自定义静默时段 |

## 十三、验收标准（核心场景）

### 13.1 课表查询

- 用户问「下节课是什么」，3 秒内返回课程名、地点、时间
- 课表未导入时，引导用户走 OCR 或手动录入，不报错中断

### 13.2 死线提醒

- 登记考试后，D-7 / D-3 / D-1 / D-0 四个节点准时推送，内容与阶梯表一致
- 阶梯提醒语气逐级加重，可被用户感知

### 13.3 计划生成与编排

- 生成的计划必须避开课表上课时段，每天时长不超过设定上限
- 一次对话中完成「登记死线 + 识别课表空隙 + 生成计划」的编排，而非分三次应答

### 13.4 边界处理

- 非学业提问不直接回答，但回复中包含当前最紧迫的死线引导
- 回复风格符合「极简、工具感、零寒暄」要求
